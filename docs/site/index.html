A abordagem

ai-squad é uma escolha de processo, não só uma coleção de prompts melhores. Vale entender o porquê antes do como — e onde isso se encaixa no espectro de jeitos de codar com IA hoje.

Spec-driven com agentes especializados

Spec-driven significa que o artefato primário é a especificação, não o código. PRD captura o problema e os critérios de aceite. Tech spec captura a solução técnica e os contratos. O código vem depois, como execução da spec — e pode ser regerado se a spec mudar.

Agentes especializados significa que cada papel é um agente diferente, com seu próprio prompt, modelo (Opus pra raciocínio, Sonnet pra execução, Haiku pra docs) e gatilho. Ninguém é generalista. Quem orquestra é uma skill (/sdlc-orchestrator) que conhece o fluxo, decide quem entra quando, e aplica os gates de qualidade.

Depois que as fases iniciais fecham (PRD + clarify gate), o orchestrator oferece um handoff via /goal que entrega o restante do fluxo de forma autônoma até o merge — pra quando você quer parar de aprovar stage por stage. A autonomia só cede em manual-only actions (signups, OAuth) ou tensão com princípios da vision do projeto.

Onde o humano entra

Autonomia agentic não significa "humano fora do loop". O Tech Lead (você) é o approver em pontos específicos — não micro-gerencia, mas decide nos checkpoints onde julgamento humano importa. Tudo o que não está marcado aqui, os agentes resolvem entre si.

Modelo de maturidade — onde isso se encaixa

Pensando em IA aplicada a desenvolvimento de software como uma escala de maturidade (inspirado em CMMI, mas adaptado pra o contexto agentic), dá pra ver 4 níveis. A progressão não é "L1 ruim, L4 bom" — é "qual nível serve qual caso". ai-squad é o ferramental pra operar no L4 quando o problema pede.

Nível 1

Vibecoding

prompt-and-pray
O que é
IA gera código por intuição a partir de prompt solto. Sem spec, sem revisão sistemática, sem testes. Você aceita o que vier.
Benefícios colhidos
Velocidade máxima em exploração. Zero fricção de processo. Ideal pra protótipo descartável, script solo, weekend hack.
Nível 2

AI-assisted dev

copilot + chat
O que é
IDE com tab completions (Copilot, Cursor) e chat lateral pra snippets. Humano segue como driver principal; IA assiste localmente.
O que muda do N1
IA deixa de ser autora solo. Vira assistente local sob revisão humana. Cada sugestão passa pelo seu olho antes de virar código.
Benefícios colhidos
Velocidade individual sem perder controle arquitetural. Code review continua sendo humano. Funciona em qualquer codebase sem setup.
Nível 3

Single-agent execution

um agente, um prompt
O que é
Um agente generalista (Claude Code, Cursor agent mode) implementa feature inteira a partir de prompt detalhado. Humano revisa o diff final.
O que muda do N2
IA volta a ser autora — mas agora com contexto multi-arquivo e iteração agentic (roda tools, lê código, testa). Você descreve a feature, ela vai e faz.
Benefícios colhidos
Features inteiras entregues por prompt. Tech Lead deixa de escrever código repetitivo — vira revisor. Salto grande de velocidade em features de complexidade média.
Nível 4 você está aqui

Multi-agent SDD

spec-driven orchestrated
O que é
Agentes especializados por papel (PM, designer, arquiteto, eng, QA, security, docs). Orquestrador aplica gates. Retrospective evolui o próprio processo.
O que muda do N3
Especialização em vez de generalista. Gates automáticos (security, perf, QA, docs) embutidos no fluxo. Spec como artefato primário antes de código. Aprendizado entre módulos via retrospective.
Benefícios colhidos
Código long-lived produzível por agentes sem virar dívida. Disciplinas garantidas (não opcionais). Sistema fica mais rápido módulo a módulo. Multi-pessoa consegue colaborar sobre o mesmo código.

Detalhe importante: você não precisa escolher um nível e ficar nele. Time/projeto maduros usam L4 pro código de produção e L1 pra hackar um script de migração que vai rodar uma vez. A maturidade é em saber quando usar qual.

# PRD: Comentários em tasks

## Problema
Discussão sobre tasks acontece fora do produto (Slack, email) e o
contexto se perde quando alguém entra na task semanas depois.

## Solução (alto nível)
Thread de comentários inline em cada task, visível pra todos que
têm acesso à task.

## Non-goals (v1)
- Notificações (push/email) — fora do escopo
- Menções @user — fora do escopo
- Anexos — fora do escopo
- Reactions/emoji — fora do escopo

## Critérios de aceite
✓ Usuário autenticado pode postar comentário (1–2000 chars)
✓ Comentário aparece imediatamente na thread (otimista)
✓ Apenas o autor pode editar/deletar seu próprio comentário
✓ Comentários ordenados cronologicamente (mais antigo primeiro)
✓ Markdown básico suportado: bold, italic, links
✓ Comentário deletado mostra "comentário removido pelo autor"
   (mantém o slot, não some)

## Métrica de sucesso
30% das tasks ativas têm ≥ 1 comentário em 30 dias.

# UX Spec: Comments thread

## Onde mora
Aba "Discussão" na página de task, à direita do form principal.
Em viewport < 768px: vira accordion abaixo do form.

## Estados
- Empty: "Nenhum comentário ainda. Seja o primeiro."
- Loading: 3 skeletons (avatar + 2 linhas)
- Error: banner vermelho "Não consegui carregar.
                                  Tentar de novo →"
- Posted: aparece otimista; se falha, undo banner

## Componentes (reusa do DS)
- <Avatar size="sm"> — autor
- <Card> — container do comentário
- <Textarea> — campo de input (auto-grow, max-h 200px)
- <Button variant="primary" size="sm"> — "Comentar"
- <DropdownMenu> — ações (editar, deletar) só pro autor

## Copy
- Placeholder: "Escreva um comentário…"
- Botão primário: "Comentar"
- Tooltip editar: "Editar comentário"
- Confirmação delete: "Apagar este comentário? Não dá pra desfazer."

## Acessibilidade
- Thread: <section aria-label="Comentários">
- Cada comentário: <article>
- Edição: focus management volta pro textarea após salvar
- Markdown render: alt em links, no autoplay em mídia
- Contrast: AA mínimo (testar "(editado)" tag)

# Tech Spec: Comments — T2

## API contracts
POST   /api/tasks/:taskId/comments   { body: string }      → 201 { id, ... }
GET    /api/tasks/:taskId/comments                          → 200 [ ... ]
PATCH  /api/comments/:id              { body: string }      → 200 { id, ... }
DELETE /api/comments/:id                                    → 204

Auth: Bearer token (existing middleware). Owner-check no PATCH/DELETE.

## DB schema (migration)
CREATE TABLE comments (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  task_id     UUID NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  author_id   UUID NOT NULL REFERENCES users(id),
  body        TEXT NOT NULL CHECK (char_length(body) BETWEEN 1 AND 2000),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  edited_at   TIMESTAMPTZ,
  deleted_at  TIMESTAMPTZ  -- soft-delete
);
CREATE INDEX idx_comments_task_id ON comments(task_id, created_at);

## Validation & security
- Body sanitization: server DOMPurify antes de salvar
- Markdown render: client (marked) c/ allowlist tight
- Rate limit: 30 comentários/min/user (existing middleware)
- XSS: body sempre em <p> com escape default no React

## Observabilidade
SLI:    P95 POST latency < 200ms
SLO:    99.5% requests < 500ms (rolling 7d)
Alert1: burn — SLO burn rate > 14.4 (1h window)
Alert2: symptom — error rate > 1% (5min window)
Events: comment.posted, comment.edited, comment.deleted

## Mapa de delegação
backend-engineer:
  - migration + repo + 4 endpoints + tests (Jest)
frontend-engineer:
  - hook useComments + 4 componentes (Thread, Item, Form, Empty)
  - integração c/ existing TaskPage

Artefatos do fluxo — referência

Você viu vários deles no exemplo acima (PRD, UX spec, tech spec, ADR). Esta é a lista completa do que circula em cada estágio: propósito, quem escreve, formato e onde mora no projeto.

📄

PRD

O contrato do produto. Problema, solução high-level, critérios de aceite testáveis, non-goals, métricas de sucesso.
Quem escreve: product-manager
Formato: Markdown estruturado (problema → solução → AC → métricas)
Localização: docs/specs/<módulo>/PRD.md
🎨

UX Spec

Como a feature aparece pro usuário. Layout, estados (vazio/loading/erro/sucesso), copy, acessibilidade, inventário de componentes do DS reusados vs novos.
Quem escreve: product-designer
Formato: Markdown + referências ao design system
Localização: docs/specs/<módulo>/ux-spec.md
🏗️

Tech Spec

A solução técnica. API contracts, schema DB, validação, segurança, observabilidade (SLI/SLO + alertas), mapa de delegação. Profundidade calibrada ao tier (T1 delta / T2 padrão / T3 deep).
Quem escreve: software-architect
Formato: Markdown estruturado + diagramas Mermaid
Localização: docs/specs/<módulo>/tech-spec.md
⚖️

ADR

Architecture Decision Record. Captura trade-offs materiais: contexto, decisão, alternativas consideradas, consequências. Imutável depois de aceito (decisão nova vira ADR novo).
Quem escreve: software-architect
Formato: Markdown (template MADR — contexto/decisão/consequências)
Localização: docs/adr/<NNNN>-<slug>.md
🎨

Design System

A fundação visual. Cores, tipografia, espaçamentos, componentes base — fonte da verdade pra todas as telas. Escrito 1x por projeto (Design System Mode) ou extraído (Documentation Mode em brownfield).
Quem escreve: product-designer
Formato: Markdown + tokens estruturados (cor/space/type)
Localização: docs/design-system.md
📘

CLAUDE.md

O contexto perene do projeto. Stack, convenções, restrições, bloco ## Tooling com integrações externas. Todo agente lê antes de qualquer ação — sem ele, alucinam convenção.
Quem escreve: humano (greenfield) ou software-architect (brownfield discovery)
Formato: Markdown — seções fixas (Stack, Convenções, Tooling, NÃO fazer)
Localização: ./CLAUDE.md (raiz)
📊

OpenAPI

O contrato de API formalizado. Atualizado pelo tech-writer após implementação. Permite contract-testing, geração de SDK, e mocks pra frontend antes do backend pronto.
Quem escreve: tech-writer
Formato: YAML/JSON (OpenAPI 3.x)
Localização: docs/openapi.yaml
📈

Maturity Assessment

Onde a squad está hoje. Rubrica 5×4 (5 dimensões × 4 níveis: Ad-hoc → Otimizando). Baseline auto-reivindicado no onboarding; revisado nas retrospectives.
Quem escreve: sdlc-orchestrator (template) + humano (claim)
Formato: Markdown — matriz de rubrica
Localização: docs/maturity-assessment.md

Princípio: spec é contrato vivo. Quando a implementação revela que a spec estava errada, atualiza a spec primeiro, depois o código — nunca o contrário. Drift entre spec e código é dívida que aparece no consistency-check gate.

Onde isto não brilha

ai-squad é uma escolha de processo — tem trade-offs reais. Saber quando não usar é parte da maturidade. Honest list do que ainda dói ou exige cuidado:

⚖️

Overhead em tarefas triviais

Pra hotfix de 1 linha ou script que vai rodar uma vez, o overhead engole o ganho. T1 mitiga pulando etapas, mas mesmo T1 é mais processo que vibecoding cru. Pra exploração rápida ou prototipagem solo, vibecoding ainda é o tool certo.

📘

Depende de CLAUDE.md decente

Sem o arquivo de contexto bem preenchido, agentes alucinam convenção. Greenfield exige curadoria manual; brownfield exige rodar /onboard-brownfield primeiro. Atalho aqui = retrabalho depois.

🧠

Não substitui domínio

Regras de negócio específicas, edge cases do produto, restrições regulatórias — continuam sendo seu trabalho documentar no PRD e na tech spec. Agentes implementam, não inventam regras. Garbage in spec = garbage out code.

🔌

Single point of failure: Claude API

API offline = squad paralisada. Sem fallback automático pra outro modelo. Pra equipes em ambiente regulado/air-gapped, isto pode ser bloqueador.

🌀

Loops, divergência, alucinação

Agentes podem entrar em loop, divergir do escopo, ou inventar API que não existe. Gates automáticos pegam a maioria (security bloqueia, consistency-check pega drift, QA pega regressão). O resto depende do humano interromper — checkpoints existem por isto.

O que vem por aí

O processo é vivo — evolui módulo a módulo via retrospective. Algumas frentes específicas estão no radar agora pra serem incorporadas como disciplina permanente no fluxo.

🔭

Agente de Operação (pós-deploy)

Hoje o fluxo termina no merge — o que acontece no ambiente em produção depois disso fica fora do escopo dos agentes. Falta um agente que assume daí: monitora o ambiente, garante que os SLIs/SLOs definidos na tech spec estão sendo respeitados, dispara alerta antes de virar incidente, e fecha o ciclo do contrato de observabilidade.

Possível forma: agent novo (operations-engineer) ou modo operate no cloud-architect
Conecta com: observability contract da tech spec (SLI/SLO/alerts), incident-commander (futuro), retrospective gate
📊

Feature nasce com métricas de uso

Toda nova feature deveria sair do berço já instrumentada: event schema definido, active users tracking, adoption funnel, retention por feature. Hoje isso é manual, fica pra depois — e na prática raramente vem. Vira disciplina embutida no PRD (KPIs) e na tech spec (event schema).

Toca: product-manager define KPIs no PRD, software-architect define event schema na tech spec, tech-writer documenta os eventos
Conecta com: bloco ## Tooling > product_analytics no CLAUDE.md (PostHog/Mixpanel/Amplitude)
🌪️

Chaos engineering no fluxo

Sistemas distribuídos só são resilientes se você prova — caso contrário, "resiliência" é wishful thinking. Vira disciplina embutida em 3 pontos: no processo (gate de chaos test antes de promover pra prod), na spec (cenários de falha declarados no tech spec — o que acontece se DB cair? cache morrer? upstream timeout?), e no gateway (fault injection controlada via Sensedia / Envoy / equivalente, com blast radius limitado).

Toca: software-architect declara cenários de falha na tech spec, quality-architect define chaos suite, cloud-architect provisiona fault injection no gateway
Conecta com: contrato de observabilidade (SLO burn precisa pegar a falha injetada), retrospective gate (cada incidente real vira chaos scenario na suite)

Outras ideias persistidas em docs/improvements.md — 17 itens organizados em T1/T2/T3 cobrindo eval suites, telemetria de invocação, dry-run mode, novas disciplinas (release engineering, incident commander, FinOps), retrospective inter-módulos, onboard-greenfield. Este doc mostra só as 2 frentes que estão no radar pra ganhar status de disciplina permanente.

Vários safeguards em camadas:

1. Gates automáticos pegam. Security bloqueia merge se HIGH finding. Consistency-check pega divergência da spec. QA pega regressão funcional. Cold-reader pega doc ruim.

2. Tech Lead aprova em checkpoints específicos (start, PRD, specs, security, consistency, merge, retrospective). Pode reverter ou interromper a qualquer momento — Ctrl+C funciona.

3. Retrospective gate captura erros pra virar diff nos agentes na próxima vez. Mesmo erro raramente acontece duas vezes.

Cenário pior: agente em loop. Você interrompe; orchestrator pede clarify ou reverte pro último checkpoint estável.

Esses ferramentas sentam no Nível 3 do modelo de maturidade: um agente generalista executa feature inteira via prompt detalhado, humano revisa diff final. São excelentes pro que se propõem.

ai-squad é Nível 4: vários agentes especializados por papel (PM, designer, arquiteto, eng, QA, security, docs), gates automáticos embutidos, retrospective evolui o sistema, spec como artefato primário.

Trade-off honesto: ai-squad é mais lento e mais caro em features triviais. Em código long-lived multi-pessoa, o overhead paga em manutenção previsível, dívida controlada, e disciplinas garantidas.
